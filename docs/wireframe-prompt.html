Role e Contexto do Produto

Instrução para a IA: Você é um senior UI engineer. Sua tarefa é produzir um único arquivo HTML auto-contido que sirva como wireframe interativo para o aplicativo web Corelix. O wireframe deve estar

pronto para evoluir para um app React em produção: cada decisão de layout, unidade de espaçamento e classe CSS deve ser compatível com a stack final descrita neste documento.

Sobre o Produto

Corelix é um SaaS de secretária digital para profissionais autônomos de saúde e bem-estar no Brasil (psicólogos, fisioterapeutas, nutricionistas, personal trainers). Combina gerenciamento de agenda, cadastro de clientes, relatórios financeiros e comunicação via WhatsApp com assistência de IA.

Insight-chave para decisões de layout

Ao abrir o app de manhã, Ana precisa responder quatro perguntas imediatamente:

1
"Quem vou atender hoje e quando?"
A agenda do dia é o elemento hero — deve estar visível acima do fold sem scroll.
2
"Tem mensagem urgente no WhatsApp?"
A IA monitora as conversas e notifica quando uma precisa de atenção. A plataforma exibe um mini-feed de status; o profissional responde diretamente pelo seu próprio WhatsApp (celular ou WhatsApp Web) — sem chat dentro da plataforma.
3
"O dia está financeiramente no caminho certo?"
KPI de receita estimada do dia (sessões concluídas × valor da sessão) com meta e progresso.
4
"É fim do mês?"
Alerta contextual quando o fim do mês se aproxima: lembra o profissional de obrigações pendentes como emissão de notas fiscais, envio de cobranças e relatórios de acompanhamento para clientes.

Stack Final e Requisitos de Compatibilidade

Atenção: O wireframe em HTML puro deve ser totalmente compatível com esta stack. Cada decisão tomada aqui (nomes de classes, estrutura de componentes, espaçamento) será migrada diretamente para o frontend React final.

Design System de Referência: Este wireframe usa o Design System "Glass/Manrope" como base visual. A IA que gerar o wireframe deve seguir os tokens e componentes descritos na Seção 3, que incluem: fundo #cfddea, superfícies glass (bg-white/50 backdrop-blur-md), primária preta, tipografia Manrope nos títulos, cards rounded-2xl, seções rounded-3xl, animações animate-slide-up e botões rounded-full.

Regras de compatibilidade para o wireframe HTML

RegraImplementação no wireframe
Tailwind via CDN Usar classes Tailwind CSS v3 via CDN — idênticas às v4 na prática
HTML semântico Sem div soup — usar <nav>, <main>, <header>, <aside>, <section>, <article>
Anotação de componentes Todo elemento relevante com data-component="NomeDoComponente" para mapeamento React
CSS custom properties Apenas tokens semânticos — nunca hex hardcoded inline. Ex: var(--color-primary)
Unidade de espaçamento Base de 4px (alinhado ao Tailwind: 1 unit = 4px)
Border radius Cards: rounded-2xl (1rem = 16px). Seções glass: rounded-3xl (1.5rem = 24px). Botões/pills: rounded-full. Tudo via variáveis var(--radius-lg), var(--radius-xl), var(--radius-full).
Zero dependências externas Apenas Tailwind CDN. Sem React, sem libs JS

Design Tokens

Design System: Todos os tokens abaixo derivam do Design System "Glass/Manrope". A IA que gerar o wireframe deve definir exatamente este bloco :root no <style> do arquivo gerado.

Defina todos os tokens no bloco <style> do wireframe. Nenhum valor de cor deve aparecer hardcoded fora deste bloco.

Animações de entrada obrigatórias

Definir no <style> do wireframe. Aplicar com delays escalonados para elementos hero:

@keyframes fadeIn  { to { opacity: 1; } }
@keyframes slideUp { to { opacity: 1; transform: translateY(0); } }

.animate-fade-in  { animation: fadeIn  0.8s ease-out forwards; opacity: 0; }
.animate-slide-up { animation: slideUp 0.8s ease-out forwards; opacity: 0; transform: translateY(20px); }
.animate-delay-200 { animation-delay: 0.2s; }
.animate-delay-400 { animation-delay: 0.4s; }
.animate-delay-600 { animation-delay: 0.6s; }
.animate-delay-800 { animation-delay: 0.8s; }

Requisitos de Entrega do Wireframe

Instrução final para a IA: Gere apenas HTML + CSS inline + vanilla JS mínimo. Sem React, sem frameworks. O arquivo deve ser auto-contido e funcionar ao abrir localmente no browser.

  • Design System "Glass/Manrope" obrigatório — fundo #cfddea, superfícies bg-white/50 backdrop-blur-md rounded-3xl, primária preta, font Manrope nos títulos, cards rounded-2xl, botões rounded-full — exatamente como especificado na Seção 3
  • Arquivo único — todo CSS no <style>, JS de navegação inline no <script>
  • Navegação interativa — clicar nos itens da sidebar mostra/oculta a tela correta (toggle de classes JS, sem frameworks)
  • Conteúdo em português real — sem lorem ipsum, sem texto em inglês exceto labels técnicos
  • Todas as 6 telas implementadas — Início, Agenda, Clientes, WhatsApp, Relatórios, Configurações
  • Anotação de componentes — data-component="NomeDoComponente" em todo elemento relevante para migração React
  • Responsivo — desktop (≥1024px) e mobile (<768px) devem funcionar sem scroll horizontal
  • Sem dependências externas — apenas Tailwind CSS via CDN
  • Toggle "Com dados / Vazio" — botão na toolbar do wireframe para alternar entre estado preenchido e empty state
  • CSS custom properties apenas — nenhum hex hardcoded fora do bloco :root
  • Hierarquia de prioridade visual — os elementos PRIORITY 1 devem ter maior peso visual (tamanho, posição, contraste)

Como usar este documento: Copie todo o conteúdo das Seções 1 a 12 e cole como prompt no Claude.ai, ChatGPT-4o ou v0.dev. Para o v0, adicione ao início: "Generate only HTML + inline CSS + minimal vanilla JS. No React."
